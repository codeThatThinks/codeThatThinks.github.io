Linux GUI Toolkit

Every toolkit that is available now sucks in one way or another. There is a definite need for an awesome toolkit that can become the single best toolkit for writing graphical Linux applications, so much so that we can abandon all other toolkits and end up with a pretty consistant user experience across all applications and Linux platforms. We need common sense merged with toolkit design.

What makes a toolkit awesome, you ask? Well, it probably should:

  • Target Linux only
    When using MVC, the frontend for an application can easily be switched out for the native GUI library for another platform quite easily if the core the application is written well. Only then can a Linux toolkit focus solely on providing a good user experience without the need for os portability.
  • Be windowing system agnostic
    A toolkit should be written in a modular way that allows it to support a variety of windowing systems and that support for new protocols can be added easily, such as in the case of switching from Xorg to Wayland. This should work similarly to how applications handle multiple frontends with MVC.
  • Be window manager agnostic
    It is extremely irritating when an application using a specific toolkit is designed only for a floating window manager and ceases to function properly when using on other types of window managers. Standards for window managers provide clients with the information neccessary to determine what behavior a toolkit should provide in order to be compatible with the window manager. Any good toolkit must follow window manager standards and adjust accordingly.
  • Take advantage of the GPU
    Graphical Processing Units are supposed to handle graphics related tasks, and it would be ridiculous if a Graphical User Interface toolkit didn't use the GPU. In some cases the GPU can be used to leverage performance while still providing an interface that is pleasing to the eye.
  • Have a simple programming interface
    A toolkit should be both easy to use and easy to understand from a programming perspective. Whether or not this involves OO programming or something entirely different altogether, like functional programming, is unknown. Regardless, an awesome toolkit shouldn't have some kind of fake OO poorly bolted to an imperative language like C (cough, cough, GTK).
  • Provide a modern user experience
    It seems like this would be an obvious goal for a toolkit, however many toolkits act like it's still the early 90s. An awesome toolkit should have some of the hipster-esque widgets such as segmented buttons, popovers, and token text fields, as well as modern behaviors such as universal drag and drop and autocorrect.

It might be impossible for an open source toolkit to implement all these things, but we're already part way there. Many toolkits provide some of these things, and with the freedom that comes with FLOSS, creating a toolkit that steals parts from these other toolkits is actually possible. Please someone (or some people) go do this.
